# Learning Goals — 学习目标管理

## 触发条件

当用户表达学习意愿时触发，例如：
- "我想学..." / "帮我规划..." / "准备考试..." / "掌握 XXX 技能"

## 目标分类

Agent 通过一个问题判断目标类型：**"学了之后怎么算学会？有没有考试/证书/等级标准？"**

| 类型 | 特征 | 示例 | 验收标准 |
|------|------|------|---------|
| **有标准型** | 有外部验收标准（考试大纲/等级框架/证书要求） | 考中级经济师、IELTS 7.0、HSK 6 级、CFA 二级 | 考试大纲/官方要求 |
| **无标准型** | 没有明确外部验收标准，"学会"的定义需要自己定 | 掌握 LLM 原理、熟练使用 Excel、学会摄影、学吉他 | **和用户共同定义里程碑** |

> 注意：IELTS/HSK/JLPT 等语言等级考试属于**有标准型**（有官方大纲和分数线）。只有"想流利说英语但不考证"才是**无标准型**。

---

## 有标准型目标流程

### 第 1 步：初步了解
1. **考什么/学什么？**（具体名称）
2. **什么时候考/什么时候要达标？**
3. **当前基础如何？**

### 第 2 步：获取验收标准
- 联网搜索该考试/等级的官方大纲和要求
- 提取：考试内容范围、通过分数线/等级标准、考试日期
- 将大纲作为知识树生成的重要依据

### 第 3 步：确认并保存

```yaml
goals:
  - goalId: "goal-001"
    type: "benchmarked"                   # 有外部标准
    subject: "科目名称"
    description: "用一句话描述学习目标"
    why: "学习动机"
    baseline: "当前基础"
    benchmark:                            # 验收标准
      source: "中级经济师考试大纲 2026"
      passingScore: 84
      examDate: "2026-11-01"
      syllabusRef: "https://..."          # 大纲链接（如有）
    targetMastery: 4
    deadline: "2026-10-25"                # 学习截止日（考前留出冲刺时间）
    dailyMinutes: 60
    createdAt: "2026-06-06"
    status: "active"
```

## 无标准型目标流程

核心问题：**帮用户把"掌握"翻译成"能做到什么"**。

### 第 1 步：了解使用场景

问用户：
1. **为什么想学？**（工作需要/个人兴趣/技术储备）
2. **学了之后想做什么具体的事？**（越具体越好）
3. **当前基础如何？**

> "掌握 LLM 原理" → "想读懂论文、能跑通微调、能帮团队做技术选型"
> "熟练使用 Excel" → "能独立做月度报表、能做 Dashboard、能自动化重复操作"

### 第 2 步：定义里程碑（核心步骤）

和用户一起定义 3-5 个**由浅到深的里程碑**。Agent 先联网检索该领域的通用能力模型，提出建议，用户确认或修改。

**每个里程碑必须满足：**
- **有具体行为**：能做什么（不是"理解"，而是"能解释/能实现/能应用"）
- **可自我检验**：用户自己能判断是否达到
- **有时间节点**：预计什么时候完成

**建议框架**：按"认知→应用→综合→创造"递进

```
层次 1 — 能解释（认知）
  "能用自己的话向别人解释 XXX 的核心概念"

层次 2 — 能操作（应用）
  "能独立完成 XXX 的标准操作/流程"

层次 3 — 能解决（综合）
  "遇到 XXX 相关问题时，能独立分析并提出解决方案"

层次 4 — 能创造（创造）（可选）
  "能基于 XXX 独立完成一个项目/作品"
```

Agent 根据**科目特性**建议不同的里程碑粒度：

| 科目特性 | 里程碑风格 | 示例 |
|---------|-----------|------|
| 编程/技术类 | 项目驱动 | "跑通 demo" → "完成一个小项目" → "解决实际问题" |
| 语言类（不考证） | 场景驱动 | "能点餐" → "能日常对话" → "能讨论工作" → "能看无字幕电影" |
| 设计/艺术类 | 作品驱动 | "临摹作品" → "独立创作" → "形成个人风格" |
| 理论/知识类 | 输出驱动 | "能解释概念" → "能写读书笔记" → "能给他人讲解" |
| 工具/软件类 | 任务驱动 | "完成基础操作" → "独立完成工作流" → "自动化重复任务" |

#### 项目驱动型（编程/技术类）的项目引导

当目标是项目驱动型时（"想学 Python"、"想学 Web 开发"），**用户通常不知道做什么项目**。Agent 必须主动引导，而不是让用户自己想。

**引导流程：**

1. **了解用户的工作场景和兴趣方向**
   - "你工作中接触最多的是什么领域？"（电商/金融/数据/自动化/…）
   - "学了之后最想解决的第一个问题是什么？"
   - "有没有一直想做但不会做的东西？"

2. **联网检索 + 匹配推荐**
   - 搜索该领域的**入门级项目清单**（GitHub trending repos、技术社区推荐项目、教程类项目的 README）
   - 根据用户的工作场景/兴趣，从搜索结果中挑选 3-5 个**具体项目**推荐给用户
   - 每个项目标注难度（入门/进阶/综合）和预计完成时间

3. **展示推荐项目（必须有具体描述，不是泛泛而谈）**

   错误示例： "做一个 Web 应用"
   正确示例：✅ "做一个个人记账工具：命令行输入收支，SQLite 存储，月底自动生成分类统计报表"

   | # | 项目名称 | 核心功能 | 用到什么知识 | 预计 |
   |---|---------|---------|-------------|------|
   | 1 | 命令行记账工具 | 输入收支→SQLite 存储→月度统计报表 | 基础语法、文件 IO、数据库 | 2 周 |
   | 2 | 网页爬虫 + 数据分析 | 爬取某网站数据→清洗→可视化图表 | 爬虫、数据处理、图表库 | 3 周 |
   | 3 | 简易 REST API | 用 Flask/FastAPI 搭建一个待办事项 API | Web 框架、数据库、HTTP | 4 周 |

4. **用户选择后，定义"完成"的标准**
   - 不是"做完就行"，而是明确交付物
   - 例：记账工具 → "能从命令行输入 10 条收支记录，运行 `python report.py` 输出本月分类统计"

5. **项目可以替换**：学到中途用户发现项目不合适，随时换一个新项目，已有知识不浪费

**示例：「3 个月内掌握 LLM 原理」（理论+技术类）**

| 里程碑 | 验收标准 | 预计 | 验收方式 |
|--------|---------|------|---------|
| M1 | 能向同事解释 Transformer、Attention 的基本原理 | 第 2 周 | 口头讲解 + Agent 提问 |
| M2 | 能读懂 HuggingFace 官方文档，跑通推理 demo | 第 5 周 | 屏幕共享演示 |
| M3 | 能读懂 LLM 论文方法章节，讲清创新点 | 第 8 周 | 论文精读 + Agent 提问 |
| M4 | 能用 LoRA 完成小模型微调，解释关键参数 | 第 11 周 | 实操演示 + Agent 提问 |
| M5 | 面对业务场景能给出 LLM 方案选型建议 | 第 12 周 | 场景模拟 + Agent 评估 |

**示例：「3 个月学会 Python」（编程类，项目驱动）**

用户背景：做电商运营，日常处理大量 Excel 数据，想自动化重复操作。

Agent 推荐项目后，用户选择了「命令行记账工具」：

| 里程碑 | 验收标准 | 预计 | 验收方式 |
|--------|---------|------|---------|
| M1 | 能写出完整脚本：从命令行输入一条收支记录并存入 SQLite | 第 1 周 | 代码运行 + 讲解 |
| M2 | 能查询和筛选记录，命令行输出指定月份的收支明细 | 第 2 周 | 代码运行 |
| M3 | 能生成月度分类统计报表（分类汇总 + 百分比），用文字表格展示 | 第 3 周 | 代码运行 + 讲解逻辑 |
| M4 | 能用 matplotlib 把月度报表画成饼图和柱状图 | 第 4 周 | 代码运行 + 截图 |
| M5 | 打包成一个独立工具：支持增删查改 + 报表生成 + 图表导出，给别人用不需要懂代码 | 第 6 周 | 实操演示 + 代码审查 |

**示例：「3 个月流利英语口语」（语言类，不考证）**

| 里程碑 | 验收标准 | 预计 | 验收方式 |
|--------|---------|------|---------|
| M1 | 能用英语完成日常场景（点餐/购物/问路） | 第 3 周 | 角色扮演对话 |
| M2 | 能用英语描述工作经历和兴趣爱好（3分钟不间断） | 第 6 周 | 口语录音 + Agent 评估 |
| M3 | 能参与英文工作会议并表达观点 | 第 9 周 | 模拟会议 + Agent 提问 |
| M4 | 能无字幕看懂英文技术视频并复述要点 | 第 12 周 | 看视频 + 口头复述 |

### 第 3 步：确认并保存

```yaml
goals:
  - goalId: "goal-002"
    type: "unbenchmarked"                 # 无外部标准
    subject: "LLM 原理"
    description: "3 个月内掌握 LLM 核心原理，能读懂论文、跑通微调、做技术选型"
    why: "工作技术储备，为团队引入 LLM 方案做准备"
    baseline: "有编程基础，了解 ML 基本概念，未接触过 Transformer"
    targetMastery: 4
    deadline: "2026-09-15"
    dailyMinutes: 60
    milestones:                           # 无标准型专属
      - milestoneId: "m1"
        level: 1
        title: "理解核心概念"
        criteria: "能向同事解释 Transformer、Attention、预训练的基本原理"
        targetDate: "2026-07-01"
        status: "pending"
        assessmentMethod: "口头讲解 + Agent 提问检验"
      - milestoneId: "m2"
        level: 2
        title: "读懂技术文档"
        criteria: "能独立读懂 HuggingFace 官方文档，跑通推理 demo"
        targetDate: "2026-07-22"
        status: "pending"
        assessmentMethod: "屏幕共享演示 + Agent 追问"
      # ... 更多里程碑
    createdAt: "2026-06-15"
    status: "active"
```

### 里程碑验收

每个里程碑到期时，Agent 通过**实际检验**确认用户是否达标：

| 验收方法 | 适用场景 | 示例 |
|---------|---------|------|
| **口头讲解** | 概念理解类 | "用自己的话解释 Attention 机制" |
| **实操演示** | 操作技能类 | "现在打开 Excel 做一个数据透视表" |
| **代码运行** | 编程类 | "跑通微调脚本，解释输出结果" |
| **角色扮演** | 语言类 | "我们来模拟一个餐厅点餐场景" |
| **场景模拟** | 综合应用类 | "假设老板让你选型 LLM，你怎么做？" |
| **作品提交** | 创造类 | "提交你做的 Dashboard / 项目 / 方案" |
| **录音/录像** | 表演/语言类 | "录一段 3 分钟英文自我介绍" |

验收结果写入 `progress/<studentId>/milestone-results.jsonl`：
```json
{
  "milestoneId": "m1",
  "goalId": "goal-002",
  "studentId": "<studentId>",
  "assessedAt": "2026-07-01T20:00:00+08:00",
  "method": "口头讲解 + Agent 提问检验",
  "passed": true,
  "score": 4,
  "notes": "Transformer 三要素解释清晰，但预训练数据规模的理解有偏差"
}
```

## 第 4 步：飞书权限预检（首次创建学习项目时执行）

> **核心原则：一次性申请全部权限，不随用随申请。**

检查该学生是否已有 `learning-profiles/<studentId>/feishu-mapping.yaml`：

**如果不存在 → 执行一次性授权流程：**

1. 调用 `feishu_oauth` 检查当前 token 是否有效
2. 读取 `templates/feishu-scopes.json` 获取完整权限范围列表（tenant + user 共 180+ scope）
3. 通过 `feishu_oauth_batch_auth` 一次性提交全部 scope，发送授权链接
4. 引导用户完成 OAuth 授权
5. 授权成功后，调用 `learning-feishu-sync` 初始化飞书空间结构
6. 保存 `feishu-mapping.yaml`（含 `authorizationStatus: "authorized"`、`authorizedScopes` 列表、`authorizedAt` 时间戳）

**权限范围分类说明（完整清单见 `templates/feishu-scopes.json`）：**

| 分类 | 主要 scope 前缀 | 用途 |
|------|----------------|------|
| Aily/智能体 | `aily:*` | AI 知识库、文件、会话、技能 |
| 多维表格 | `base:*` / `bitable:*` | 学习数据数据库（5 张表）的增删改查 |
| 白板 | `board:whiteboard:*` | 知识图谱可视化 |
| 卡片 | `cardkit:card:*` | 消息卡片读写 |
| 通讯录 | `contact:*` | 用户身份识别（open_id → studentId） |
| 旧版文档 | `docs:doc*` | 文档读写、评论、权限、订阅 |
| 新版文档 | `docx:document*` | 新版文档创建、块转换 |
| 云空间 | `drive:*` | 文件夹创建、文件上传下载、版本管理 |
| 即时通讯 | `im:*` | 消息推送、群管理、Pin、表情回复 |
| 电子表格 | `sheets:spreadsheet*` | 报表数据 |
| 幻灯片 | `slides:presentation*` | 学习资料展示 |
| 知识库 | `wiki:*` | 学习空间（学科目录、章节文件夹） |
| 日历 | `calendar:*`（user scope） | 学习日程提醒（user scope） |
| 任务 | `task:*`（user scope） | 学习任务管理（user scope） |
| 搜索 | `search:*`（user scope） | 文档/消息搜索（user scope） |

**如果已存在 → 跳过。** 仅检查 token 是否过期，过期则尝试刷新。

## 目标分解

当用户说"我想学 X"但 X 太大时：
- 拆解为子领域
- 询问用户想先学哪个
- 为每个子领域创建独立目标或作为同一目标的不同阶段

## 目标修改

用户可以随时：
- 调整截止日期 / 修改目标掌握度
- 暂停/恢复某个目标
- **修改里程碑**（无标准型目标）
- 添加新的子目标

操作：读取 goals.yaml → 修改对应条目 → 写回文件

## 注意事项

- **不要替用户决定**：只提供建议，最终决定权在用户
- **有标准型**：以考试大纲/官方框架为知识树的主要依据
- **无标准型**："掌握"必须翻译成"能做到什么"，用行为描述而非抽象概念
- **里程碑是活的**：用户学到一半发现目标定高了/低了，可以随时调整
- **考虑现实性**：提醒用户时间预算是否足够，但不替用户降低目标
- **授权前置**：权限问题在学习项目创建时一次性解决
